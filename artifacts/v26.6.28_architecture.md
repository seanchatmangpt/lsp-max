## Salsa Incremental DB Integration
Incremental compilation layers have been integrated via the Salsa 0.26 incremental engine into the AST and LSIF pipelines to prevent unnecessary re-execution.

- `crates/lsp-max-adapters/lsp-max-ast/src/db.rs`: Introduces `SalsaLspAdapter` wrapping `LspMaxDb` and tree-sitter. 
  - `SourceFile` is the tracked input.
  - `ast_diagnostics` is memoized, invalidating only when `SourceFile.text` or `encoding` changes.
- `crates/lsp-max-lsif/src/db.rs`: Introduces `IncrementalLsifIndexer` wrapping `LsifMaxDb`.
  - `LsifSource` is the tracked input.
  - Caches file-level LSIF summaries (`LsifFileResult`: vertex_count, moniker_count, reference_count, has_errors).
  - Skips full re-indexing of unchanged files on subsequent runs.

## Oxigraph Boundary in semantic_graph
The control plane introduces a rigid boundary to encapsulate the Oxigraph database, preventing graph querying from polluting the hot path.

- `src/runtime/control_plane/semantic_graph/mod.rs`: Defines the `SemanticLawGraph` trait.
  - **INVARIANT: OXIGRAPH_BOUNDARY_HELD**: `oxigraph::*` is strictly confined to `store.rs` and `lsif_import.rs`.
  - **INVARIANT: OXIGRAPH_NOT_ON_HOT_PATH**: The graph is explicitly defined as COLD storage for law lookup, separated from live diagnostic computation (which uses Salsa).
- `src/runtime/control_plane/semantic_graph/store.rs`: The sole authorized implementer of `SemanticLawGraph` (`OxigraphStore`).

## LSIF Artifacts
LSIF (Language Server Index Format) generation and ingestion surfaces have been structurally established:
- `playground/lsif_dump.json`: A staging artifact representing the raw element graph emission.
- `src/runtime/control_plane/semantic_graph/lsif_import.rs`: Logic for importing LSIF JSONL data into named Oxigraph triples (bound by the cold-path invariant).
