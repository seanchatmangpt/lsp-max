## v26.6.30 — Full Claude Code LSP Client Support

**Date**: 2026-06-29  
**Status**: OPEN  
**CalVer**: 26.6.30  
**Epic Goal**: lsp-max becomes the transparent LSP proxy for every language server Claude Code connects to.

---

## Executive Summary

This epic delivers **full Claude Code LSP client integration** via a seven-ticket roadmap (CC-001 through CC-007). The compositor, currently a standalone tool for LSP multiplexing in CI and agent workflows, becomes Claude Code's primary LSP endpoint. Auto-discovery hooks scan Claude Code's active LSP connections at session boot; configuration is merged (static + auto) into a unified registry; the compositor proxies all traffic with law-state enforcement, ANDON gating, and diagnostic aggregation.

**Key architectural constraint**: The compositor must be **transparent** — Claude Code observes no behavioral difference from talking to it vs. direct child servers. Version fields are forwarded unchanged; capabilities are merged; notifications are serialized per URI; diagnostics from all servers are merged into one feed to Claude Code's `publishDiagnostics` consumer.

**Location**: `.claude/hooks/`

| Hook | Status | Purpose |
|------|--------|---------|
| `session-start.sh` | ✅ | Status report (branch, dirty files, toolchain); phase-shift bootstrap for sibling repo clones |
| `discover-lsp-chains.sh` | ⚠️ PARTIAL | Heuristic-based auto-discovery (file presence checks); writes `.claude/lsp-max-auto.toml`; **lacks** Claude Code LSP server detection, process scan, strategy 1-3 |

```

  2. discover-lsp-chains.sh    → heuristics → lsp-max-auto.toml
```

## Part V: Risk Assessment

### High Risk

1. **Per-URI Serialization Ordering (CC-004)** — Most complex; critical for correctness
2. **Claude Code Settings.json Format (CC-006)** — Depends on Claude Code internals
3. **LSIF Staleness Detection (CC-007)** — Git + filesystem integration fragility

### Medium Risk

1. **Process Scan Performance (CC-001)** — OS-specific `ps` output
2. **Probe Timeout Races (CC-002)** — Intermittent healthcheck timeouts
3. **Hot-Reload Concurrency (CC-002)** — Reloading while routing active requests

### Mitigations

- Depth limits, timeouts, graceful fallbacks
- Comprehensive unit + integration tests
- Stress testing (10,000+ URIs)
- Code review checklists
